# 0. 文档控制与阅读说明

## 0.1 文档目的

本文档用于指导“福岛岳温泉零碳宠物营地&温泉酒店”C端官方网站的信息架构、页面功能、内容策略、交互流程、数据结构、验收标准与上线范围。它不是视觉稿，也不是最终开发排期；它是产品、设计、内容、开发、运营、酒店管理团队之间的需求对齐基准。

## 0.2 本PRD的核心判断

> **产品定位一句话**
>
> 官网应被设计为“携宠家庭前往福岛岳温泉度假的官方决策入口”，而不是单纯的酒店展示页。首页必须在10秒内讲清楚：这里在哪里、能不能带宠物、有什么温泉与宠物服务、怎么预约/预订、为什么值得来。

## 0.3 资料依据与边界

- 依据项目PPT：项目为“福岛岳温泉零碳宠物营地&温泉酒店”，包含温泉酒店、宠物营地生活方式、宠物活动社群、零碳设施体验等内容。

- 依据用户补充：目前官网只面向C端用户，真实照片等素材已准备好。

- 参考站点：Warburton Lodge 的结构为旅宿官网常见结构，包括首页、历史/品牌介绍、房型、设施、服务、周边探索、预订入口与联系表单。

- 明确排除：融资信息、天使轮/A轮、B端销售、光伏树全球销售、预制房全球定制，不作为C端官网主导航内容。

- 保留方式：光伏树和预制房只在C端页面中作为“零碳营地体验”“景观设施”“宠物小别墅/营地空间”的背景亮点出现。

# 1. 项目理解与产品策略

> 把PPT中的综合商业项目收敛成C端旅宿官网

# 1.1 项目事实提炼

| 维度 | 项目资料中的事实 | C端官网转化表达 |
| --- | --- | --- |
| 空间基础 | 温泉酒店共有24间客房，12间宠物友好房，12间普通房。 | “普通温泉旅客与携宠家庭都能入住”的双客群酒店。 |
| 宠物服务 | 主人温泉与宠物专用温泉并行，配套宠物吹干、梳理、清洁、寄养、室内外Dog Run。 | 重点做成“宠物像家人一样被照顾”的服务模块。 |
| 营地体验 | 营地以小别墅式预制房为主体，包含Dog Run、BBQ、咖啡休闲、散步、运动、社交。 | 将“住一晚”升级为“携宠周末生活方式”。 |
| 活动社群 | 运动会、摄影比赛、主题派对、季节限定、宠物相亲、生日会、宠物婚礼、会员体系。 | 官网设置活动/社群页面，用于复购和二次触达。 |
| 零碳特色 | 追光光伏树在营地内作为零碳景观与能源节点，功能含照明、储能、充电、安防等。 | C端只强调“可持续度假体验”和“零碳景观”，不讲B端销售。 |
| 开业节奏 | 2026年8月日本一号营地温泉酒店模块营业。 | MVP阶段以“预约/候补名单”承接兴趣用户。 |
| 导流基础 | 动物园、Team Orange / Ebisu Circuit 可形成亲子、动物爱好者、年轻化、国际化客群导流。 | 周边探索页可设计“白天游玩+夜晚温泉住宿”的行程推荐。 |

# 1.2 C端产品定位

> **官网主张**
>
> 与宠物一起，住进福岛岳温泉的零碳周末生活方式。

C端官网的第一目标不是解释商业模式，而是让游客相信：这里真的适合携宠入住、环境舒适、服务专业、交通与周边玩法清晰，并且现在就可以预约或加入首发候补名单。

## 1.3 品牌调性

| 调性关键词 | 表达方式 | 避免方式 |
| --- | --- | --- |
| 自然 | 大图、留白、木质/温泉/山林元素、低饱和色彩。 | 避免过强科技感、过度融资感。 |
| 温暖 | 强调宠物是家人、主人放松、宠物释放天性。 | 避免冷冰冰罗列设备。 |
| 可信 | 真实照片、房型面积、服务规则、FAQ、交通方式、联系方式。 | 避免只使用概念图而缺少落地证据。 |
| 精致 | 高品质日本温泉、宠物照护、定制活动、会员体验。 | 避免低价露营地或普通民宿感。 |
| 可持续 | 零碳景观、能源节点、自然友好的低碳旅居。 | 避免把光伏树写成技术销售页。 |

# 1.5 产品目标与KPI

| 目标层级 | 目标 | 核心指标 | MVP建议目标 |

| 转化目标 | 让用户快速判断是否适合自己和宠物。 | 房型详情点击率、FAQ点击率、表单完成率。 | 房型页到表单页CTR≥12%。 |

| 运营目标 | 减少重复咨询。 | FAQ覆盖率、客服重复问题下降。 | 上线后2周整理Top 20问题并更新FAQ。 |

# 1.6 非目标

- 不把官网做成投资路演网站。

- 不公开融资金额、股权比例、A轮计划等信息。

- 不把光伏树、预制房设置为C端主导航销售页面。

- MVP不强制上线复杂实时库存、动态价格、在线支付；若运营确认价格与库存，可进入Phase 1。

- MVP不做完整会员中心和积分体系，只做活动/会员兴趣登记。

# 2.2 核心用户任务 JTBD

- 当我想带宠物去温泉度假时，我需要确认酒店是否真正宠物友好，以便放心预订。

- 当我准备周末短途旅行时，我需要快速了解房型、价格、设施与交通，以便决定是否值得安排。

- 当我担心宠物入住规则时，我需要清楚看到限制、所需证明、清洁/赔偿规则，以便避免到店后产生冲突。

- 当我想参加宠物活动时，我需要看到活动主题、时间、名额、费用和报名入口，以便提前规划。

- 当我还不能立即预订时，我需要一个低门槛方式加入候补或订阅开业信息，以便后续被提醒。

# 2.4 端到端流程：携宠预约

1. 用户进入首页，看到携宠温泉旅居卖点和“查看宠物友好房 / 加入开业候补”按钮。

1. 用户选择入住日期、人数、房间数、是否携宠、宠物数量。

1. 系统根据是否正式开放预订进入两种状态：MVP为候补名单提交；Phase 1为库存查询。

1. 用户填写基础联系方式和宠物信息：宠物类型、犬种/猫种、体重、年龄、疫苗/狂犬证明状态、特殊需求。

1. 系统显示规则确认：牵引、清洁、损坏赔偿、宠物温泉使用、Dog Run注意事项。

1. 用户提交后收到确认页和邮件/短信/LINE通知，后台生成Lead或Booking Request。

1. 运营人员在CRM中跟进，确认价格、房型、宠物规则、支付方式。

| 方案 | 适用性 | 优点 | 注意事项 |
| --- | --- | --- | --- |
| WordPress + 酒店主题/插件 | 快速上线、内容团队易维护。 | 成本低、CMS成熟、适合旅宿官网。 | 性能和插件安全需要管理；复杂预订需谨慎。 |
| Headless CMS + Next.js/Nuxt | 品牌长期运营、多语言、性能要求高。 | 速度快、SEO强、设计自由度高。 | 开发成本更高；后台需配置。 |
| SaaS建站 + 外接表单 | 极快上线MVP。 | 成本低、无需复杂开发。 | 后续预订/多语言/SEO扩展可能受限。 |

> **建议**
>
> 若目标是先获客验证，MVP可选择“Headless CMS/WordPress + 表单/CRM + 静态页面”的轻量方案；正式开业前再接入专业酒店预订引擎或自研库存模块。

# 11.3 性能要求

| 指标 | 目标 | 说明 |
| --- | --- | --- |
| LCP | ≤2.5s目标 | 首页首屏大图必须优化。 |
| CLS | ≤0.1目标 | 图片尺寸预留，避免布局跳动。 |
| INP | ≤200ms目标 | 表单和菜单响应流畅。 |
| 图片体积 | 首屏图片≤300KB目标，图库图片按需加载。 | WebP/AVIF，保留高清查看。 |
| 移动端 | 优先保障iOS Safari/Android Chrome。 | 预约流程需单手可完成。 |
| 可用性 | 表单服务99.5%可用目标。 | 失败时提供备用联系方式。 |

# 11.4 安全与隐私

- 所有表单必须通过HTTPS提交。

- 表单增加CSRF保护、频率限制、验证码或隐形反垃圾策略。

- 只收集完成预约/服务所必要的信息；宠物健康与疫苗信息属于敏感程度较高的运营信息，需限制后台访问。

- 隐私政策必须说明收集目的、存储周期、第三方服务、用户删除/修改请求方式。

- 营销订阅必须单独勾选，不能默认同意。

- 正式支付上线前必须配置退款政策、取消政策、支付安全说明。

# 12.3 内容状态标签

14. 风险、依赖与开放问题

把不确定性显性化，避免上线后返工

# 14.1 主要风险

| 风险 | 影响 | 缓解方案 |
| --- | --- | --- |
| 开业时间或设施开放时间变化 | 用户预期与实际不符，影响信任。 | 所有时间字段CMS化；使用“预计/Planned”口径；设置公告栏。 |
| 宠物规则未确认 | 可能导致到店争议或合规风险。 | 上线前由运营/法务确认规则；表单中明确同意。 |
| 照片与实际体验不一致 | 造成投诉和退订。 | 实拍优先；示意图标注；定期更新。 |
| 预订系统过早复杂化 | 延期上线、维护成本高。 | MVP先做预约/候补，正式营业前再接库存支付。 |
| B端/融资信息混入C端 | 干扰游客决策。 | 全站信息架构明确排除；零碳内容只做体验表达。 |
| 多语言内容不同步 | 用户信息冲突。 | CMS建立翻译状态；关键规则统一源。 |
